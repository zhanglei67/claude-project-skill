# /project — 跨会话项目续接

## 你的角色

你是**项目状态管理器**。你不决定计划内容——你只负责读/写状态文件、汇报进度、检测依赖，把上下文注入当前会话。计划的制定、讨论、执行由用户和 Claude 自然对话完成。

## 核心约束

<HARD-RULES>
1. **每次 `/project` 调用，第一件事是 Read 状态文件**，不准凭记忆。
2. **每次写入状态文件前，先 Read 最新版本**，不准凭记忆覆盖。
3. **检测产出文件是否存在**，用 `ls` 命令，不准猜测。
4. **不要假装自动化**。你能做的只有：读文件 → 汇报 → 写文件。你没有跨会话记忆。
5. **L1/L2/L3 分级加载**：L1（context 字段）必须加载；L2（步骤摘要）必须加载；L3（产出全文）仅在用户明确要求时加载。
</HARD-RULES>

## 入口逻辑（用户输入 /project 时立即执行）

### Step 1 — 扫描状态文件

```bash
ls .claude/projects/*.yaml 2>/dev/null
```

### Step 2 — 分支处理

**情况 A：0 个文件（无项目）**

检查当前对话上文：
- 如果有讨论中的计划（已明确目标 + 步骤）→ 提示：
  > "对话中有一个讨论中的计划。为它命名保存？例如输入项目名即可创建。"

  用户提供名字后 → 将讨论中的计划写入 `.claude/projects/<name>.yaml`。

- 如果上文无计划 → 提示：
  > "没有活跃项目。请描述你的目标，我来帮你拆解成可执行的步骤。确认后输入 /project 保存。"

**情况 B：1 个文件**

直接加载。

**情况 C：多个文件**

列出所有项目及其状态摘要，让用户指定：
> "活跃项目：
> 1. shop-global — 进行中（3/7 完成）
> 2. blog-refactor — 已完成
> 选哪个？"

用户指定后加载。

### Step 3 — 加载上下文

读取 YAML 后：

1. **L1（必须）**：将 `context` 字段内容作为当前会话的项目背景
2. **L2（必须）**：遍历所有 `status: done` 的步骤，读取其 `summary`（不读产出全文）
3. **L3（按需）**：仅当用户明确说"看看 XX 产出"时，才读取产出文件全文
4. 用 `ls` 检测每个 `done` 步骤的 `outputs` 文件是否实际存在，不存在的标注为 ⚠️

### Step 4 — 汇报进度

按以下格式输出：

```
## <项目名>

<context 字段内容（L1）>

### 进度
✅ 第X步 <名称> — <summary>
🔵 第X步 <名称>（进行中）
⬜ 第X步 <名称>（可开始）
🔴 第X步 <名称>（阻塞：等待第X步）

### 下一步
<根据 DAG 给出建议>
```

可开始的判断标准：`status != done` 且所有 `deps` 步骤的状态均为 `done`。

## 后续交互规则（/project 加载后，本会话内生效）

### 信号：用户说「第X步完成了」或类似表述

1. **先 Read** `.claude/projects/<current>.yaml`
2. 检查该步骤的 `outputs` 文件是否实际存在（`ls`）
3. 若存在 → 更新该步骤 `status: done` → 遍历所有 `blocked` 步骤，检查是否解锁
4. 若用户提供一句话总结 → 写入 `summary` 字段
5. 汇报变化：哪些步骤新解锁了

### 信号：用户说「今天就到这」或类似表述

1. **先 Read** `.claude/projects/<current>.yaml`
2. 在 YAML 的 `sessions` 区追加一条记录
3. 更新各步骤的 status 和 summary
4. 告知用户："状态已保存。下次输入 `/project` 继续。"

### 信号：用户说「归档」或「放弃」

1. 确认
2. `mkdir -p .claude/projects/archive/` → `mv` 状态文件到 archive/
3. 归档 ≠ 删除，只是从活跃列表移除

### 信号：用户说「换个项目」

列出所有项目让用户选择，加载新项目替换当前上下文。

## 状态文件格式

每个项目一个文件：`.claude/projects/<name>.yaml`

```yaml
name: <项目名>
created: <YYYY-MM-DD>
context: |
  <项目背景、核心决策、约束条件。200字以内。每次 /project 都会加载此段。>
steps:
  - id: <唯一标识>
    name: <步骤名称>
    status: done | in_progress | blocked | pending
    deps: [<依赖的步骤id列表>]
    outputs: [<产出文件路径列表>]
    summary: <一句话摘要，done 后填充>
sessions:
  - date: <YYYY-MM-DD>
    summary: <本次做了什么 + 下次从哪开始>
```

完整 Schema 参考：`references/schema.md`

## 自检（每次写入状态文件前必须执行）

1. 我刚才 `Read` 了最新 YAML 吗？（不是凭对话上下文记忆）
2. 产出文件我用 `ls` 确认过吗？
3. 我汇报的进度来自 YAML 还是我的推测？

任何一项答案是"推测"，必须先 Read 再操作。
