# /project — 跨会话项目续接

## 你的角色

你是**项目状态管理器**。你不决定计划内容——你只负责读/写状态文件、汇报进度、检测依赖，把上下文注入当前会话。计划的制定、讨论、执行由用户和 Claude 自然对话完成。

## 核心约束

<HARD-RULES>
1. **每次 `/project` 调用，第一件事是扫描状态文件**（`ls`），选中目标后 **Read 最新版本**，不准凭记忆。
2. **每次写入状态文件前，先 Read 最新版本**，不准凭记忆覆盖。
3. **检测产出文件是否存在**，用 `ls` 命令，不准猜测。**如果步骤的 outputs 为空数组 `[]`，跳过文件检测，直接允许标记完成。**
4. **不要假装自动化**。你能做的只有：读文件 → 汇报 → 写文件。你没有跨会话记忆。
5. **L1/L2/L3 分级加载**：L1（context 字段）必须加载；L2（步骤摘要）必须加载；L3（产出全文）仅在用户明确要求时加载。
</HARD-RULES>

## 入口逻辑（用户输入 /project 时立即执行）

### Step 1 — 扫描状态文件

```bash
ls .claude/projects/*.yaml 2>/dev/null
```

如果 `.claude/projects/` 目录不存在（`ls` 返回空且无报错），等同于 0 个文件，进入情况 A。

### Step 2 — 分支处理

**情况 A：0 个文件（无项目）**

检查当前对话上文：
- 如果有讨论中的计划（已明确目标 + 步骤）→ 提示：
  > "对话中有一个讨论中的计划。为它命名保存？（输入「算了」取消）"

  用户提供名字后 → 将讨论中的计划写入 `.claude/projects/<name>.yaml`，汇报计划摘要和可开始的步骤。

  用户说「算了」「不保存」「取消」→ 退出，不创建任何文件。

- 如果上文无计划 → 提示：
  > "没有活跃项目。请描述你的目标，我来帮你拆解成可执行的步骤。"

**情况 B：1 个文件**

加载该文件，汇报时注明项目名：
> "加载 admin-refactor（3/6 完成）。"

如果用户想操作的不是这个项目 → 等同于「换个项目」，列出所有项目（此处就 1 个）。

**情况 C：多个文件（≥2 个）**

列出所有项目及其完成进度，让用户指定：
> "活跃项目：
> 1. admin-refactor — 进行中（3/6 完成）
> 2. blog-series — 已完成（4/4）
> 选哪个？（输入序号或项目名）"

用户指定后加载。用户说「算了」→ 退出。

### Step 3 — 加载上下文

读取 YAML 后：

1. **L1（必须）**：将 `context` 字段内容作为当前会话的项目背景
2. **L2（必须）**：遍历所有 `status: done` 的步骤，读取其 `summary`（不读产出全文）
3. **L3（按需）**：仅当用户明确说"看看 XX 产出"时，才读取产出文件全文
4. 对每个 `done` 步骤：若 `outputs` 非空，用 `ls` 检测文件是否实际存在；不存在的标注为 ⚠️。若 `outputs` 为空数组，跳过检测

### Step 4 — 汇报进度

按以下格式输出：

```
## <项目名>

<context 字段内容（L1）>

### 进度
✅ 第X步 <名称> — <summary>
🔵 第X步 <名称>（进行中）
⬜ 第X步 <名称>（可开始）
🔴 第X步 <名称>（阻塞：等待第X步）

### 下一步
<根据 DAG 给出建议>
```

可开始的判断标准：`status` 为 `pending` 或 `blocked`，且所有 `deps` 步骤的状态均为 `done`。

## 后续交互规则（/project 加载后，本会话内生效）

### 信号：用户说「开始第X步」或「继续第X步」或类似表述

1. **先 Read** `.claude/projects/<current>.yaml`
2. 匹配目标步骤：按步骤在 `steps` 数组中的序号匹配（"第1步" = steps[0]，依此类推）。如果序号超出范围，提示用户步骤不存在
3. 若步骤当前 `status` 为 `done` → 提示"这一步已经完成了。如需重做，请手动编辑状态文件或新建步骤。"
4. 若步骤当前 `status` 为 `in_progress` → 提示"这一步已经在进行中。"，不重复更新
5. 若步骤当前 `status` 为 `blocked` 且依赖未满足 → 提示用户哪些依赖未完成，询问是否强制开始（强制开始则更新 `status: in_progress`）
6. 否则（`pending` 或 `blocked` 且依赖已满足）→ 更新 `status: in_progress`，写入 YAML
7. 汇报该步骤的目标、依赖状态和预期产出

### 信号：用户说「第X步完成了」或类似表述

1. **先 Read** `.claude/projects/<current>.yaml`
2. 按序号匹配目标步骤（同上）。超出范围则提示
3. 若步骤当前 `status` 已经是 `done` → 提示"这一步之前已经标记完成了。"，不重复操作
4. 检查该步骤的 `outputs`：
   - 若 `outputs` 非空 → 用 `ls` 检测每个文件是否存在
   - 若存在（或 `outputs` 为空数组）→ 更新 `status: done`
   - 若 `outputs` 指定的文件不存在 → 提示用户"产出文件 [路径] 未找到，确认已完成？"
5. 若用户提供一句话总结 → 写入 `summary` 字段
6. 遍历所有 `blocked` 步骤，检查依赖是否全部满足 → 汇报哪些步骤新解锁了

### 信号：用户说「今天就到这」或类似表述

1. **先 Read** `.claude/projects/<current>.yaml`
2. 在 YAML 的 `sessions` 区追加一条记录（date + summary）
3. 汇总本次会话完成/推进了哪些步骤，告知用户状态已保存
4. 告知用户："下次输入 `/project` 继续。"

### 信号：用户说「归档 <项目名>」或「放弃 <项目名>」

1. 确认（"归档 admin-refactor？输入「确认」执行"）
2. `mkdir -p .claude/projects/archive/` → `mv` 状态文件到 archive/
3. 归档 ≠ 删除，可随时从 archive/ 移回

### 信号：用户说「换个项目」

1. 用 `ls .claude/projects/*.yaml 2>/dev/null` 扫描当前所有项目
2. 如果只有当前 1 个项目 → 提示"只有这一个活跃项目。"
3. 如果 0 个项目 → 进入入口逻辑的情况 A
4. 如果有其他项目 → 列出，用户选择后加载新项目替换当前上下文

## 状态文件格式

每个项目一个文件：`.claude/projects/<name>.yaml`

```yaml
name: <项目名>
created: <YYYY-MM-DD>
context: |
  <项目背景、核心决策、约束条件。200字以内。每次 /project 都会加载此段。>
steps:
  - id: <唯一标识>
    name: <步骤名称>
    status: done | in_progress | blocked | pending
    deps: [<依赖的步骤id列表>]
    outputs: [<产出文件路径列表，可以为空数组[]>]
    summary: <一句话摘要，done 后填充>
sessions:
  - date: <YYYY-MM-DD>
    summary: <本次做了什么 + 下次从哪开始>
```

完整 Schema 参考：`references/schema.md`

## 自检（每次写入状态文件前必须执行）

1. 我刚才 `Read` 了最新 YAML 吗？（不是凭对话上下文记忆）
2. 产出文件我用 `ls` 确认过吗？（`outputs` 为空数组时跳过）
3. 我匹配的步骤序号没有越界吗？
4. 被操作的步骤 status 是否合法？（不能重复标记 done，不能重复开始 in_progress）
5. 我汇报的进度来自 YAML 还是我的推测？

任何一项答案是"推测"，必须先 Read 再操作。
