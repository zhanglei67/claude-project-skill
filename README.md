# /project — 跨会话项目管理 Skill

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Claude Code](https://img.shields.io/badge/Claude%20Code-Skill-orange)](https://claude.ai/code)

**做好分步骤计划，随时打开新会话，一条命令继续推进。不受时间和会话限制。**

---

## 为什么需要这个？

Claude Code 没有跨会话记忆。每次新会话，Claude 从零开始——不知道你之前做了什么、哪些步骤已完成、卡在哪里、产出过什么结论。

大部分需要多步骤推进的事情，都不是一次会话能做完的。今天做一部分，明天接着来。如果每次都要重新描述背景、回忆进度、翻聊天记录找上次的产出——这件事本身就消耗了你最宝贵的注意力。

`/project` 解决的就是这个：**做好分步骤计划，随时打开新会话，一条命令继续推进。** 不是记笔记，不是 Todo List——是给 AI 装了一个外置的项目记忆。

---

## 快速开始

### 安装

```bash
mkdir -p ~/.claude/skills/project
cp SKILL.md ~/.claude/skills/project/
```

或使用初始化脚本：

```bash
bash scripts/init.sh
```

### 第一次使用

```
你: "我想重构后台管理系统。跑了3年，代码臃肿。
     计划分模块做：先抽离公共组件，再逐个页面迁移，
     最后统一测试。帮我梳理步骤。"

Claude: [讨论...分析...拆解为 6 步，识别依赖关系]

你: "可以，就这个计划。"
     /project

Claude: [Skill 加载 → 检测到上文有讨论中的计划]
       "对话中有一个讨论中的计划。为它命名保存？"

你: admin-refactor

Claude: [写入 .claude/projects/admin-admin-refactor.yaml]
       "admin-refactor 已保存。6 步，预计 4-5 次会话。
        第 1-4 步无依赖可并行。从第 1 步开始？"

你: 开始第1步

--- [几天后，新会话] ---

你: /project

Claude: [扫描 → 1 个活跃项目]
       "加载 admin-refactor（2/6 完成）。

        admin-refactor · 后台管理系统重构
        第 1、2 步已完成，第 3 步进行中。
        第 5 步已解锁（依赖的第 2 步完成）。继续？"
```

**就一条命令：`/project`，不带参数。** Skill 会自动检测当前状态，有项目就续接，有讨论就提示保存，都没有就引导创建。

## 能力与边界（诚实表格）

我们对这个 Skill 的能力边界保持透明：

| ✅ 能做到 | ❌ 做不到 |
|-----------|----------|
| `/project` 唤起 → 读 YAML → 恢复项目上下文 | 在你没说 `/project` 时自动感知"该续接了" |
| 检测产出文件是否实际存在 | 记住你没写进 YAML 或产出文件的讨论细节 |
| 根据依赖关系判断哪些步骤可开始、哪些阻塞 | 自动判断产出质量是否合格 |
| 多项目并存（命名空间隔离） | 跨项目自动感知关联 |
| `/project` 加载后，后续用自然语言交互 | 在你忘记 `/project` 时强制触发 |

**核心承诺：一次 `/project`，就能在中断后的新会话里恢复项目全貌——有哪些步骤、产出什么、卡在哪、该做什么。**

## 核心卖点

### 1. DAG 编排，不只是线性笔记

```
第1步 现状分析+技术选型 ──────────────────┐
第2步 公共组件抽离 ──→ 第5步 状态管理迁移 ─┤  
第3步 用户模块迁移 ───────────────────────┤
第4步 数据报表模块迁移 ───────────────────┘
                                           ↓
                                   第6步 全量回归测试
```

普通笔记只记录"上次做到第3步了"。`/project` 会告诉你：第2步和第3步可以不同会话并行推进，第5步因为依赖第2步所以还在阻塞——谁先完成谁解锁下游。

### 2. 一条命令，零子命令

不区分 `/project continue`、`/project wrap`、`/project done`。就 `/project`，后续自然语言：

| 你说（自然语言） | Skill 响应 |
|-----------------|-----------|
| `/project` | 加载上下文，汇报进度 |
| "第1步完成了" | 锁定完成状态，检查哪些步骤新解锁 |
| "今天就到这" | 更新状态，追加会话摘要 |
| "归档这个项目" | 确认后移入 archive/ |

### 3. 多项目隔离

```
.claude/projects/
  admin-refactor.yaml
  blog-series.yaml
  side-project.yaml
  archive/              ← 已完成或放弃的项目
    old-feature.yaml
```

`/project` 只有一个活跃计划时直接加载，多个时列出让你选。项目之间完全隔离，互不影响。

### 4. 上下文分级注入

全量加载产出文件会吃掉 context window。`/project` 用三级加载：

| 级别 | 加载内容 | 触发条件 |
|------|---------|---------|
| L1 | `context` 字段（200 字项目背景） | 每次 `/project` |
| L2 | 每个 done 步骤的一行摘要 | 每次 `/project` |
| L3 | 产出文件全文 | 用户说"看看 XX 产出"时 |

### 5. 零依赖

一个 `SKILL.md` 文件 + 项目里的 `.claude/projects/*.yaml`。不需要 MCP Server、Python 脚本、API key、外部服务。

## 完整演示

### 场景：重构一个运行多年的后台管理系统（6 步，有依赖，可并行）

**Session 1 — 讨论 + 保存计划**

```
👤: "后台管理系统跑了3年多，代码臃肿，维护越来越慢。
     想重构但不敢一次全动。帮我梳理一个分模块渐进式重构的计划。"

🤖: [讨论现状 → 分析风险 → 拆解步骤]

    建议 6 步：
    第1步 现状分析+技术选型 ──────────────┐
    第2步 公共组件抽离（无依赖）──────────┤
    第3步 用户管理模块迁移（无依赖）───────┤ 三者可并行
    第4步 数据报表模块迁移（无依赖）───────┘
    第5步 全局状态管理迁移 ← 依赖第2步
    第6步 全量回归测试   ← 依赖第3、4、5步

    预计 4-6 次会话。

👤: 可以。保存这个计划。
    /project

🤖: [检测到 0 个状态文件，但上文有讨论中的计划]
    "对话中有一个讨论中的计划。为它命名保存？（输入「算了」取消）"

👤: admin-refactor

🤖: [写入 admin-admin-refactor.yaml]
    "admin-refactor 已保存。6 步，第 1-4 步无依赖可并行。
     下次在任何会话输入 /project 即可继续。从第 1 步开始？"
```

**Session 2 — 续接，并行工作**

```
👤: /project

🤖: [扫描 → 1 个活跃项目]
    "加载 admin-refactor。

     admin-refactor · 后台管理系统渐进式重构
     
     进度：
     ⬜ 第1步 现状分析+技术选型
     ⬜ 第2步 公共组件抽离
     ⬜ 第3步 用户管理模块迁移
     ⬜ 第4步 数据报表模块迁移
     🔴 第5步（等第2步）
     🔴 第6步（等3+4+5）
     
     第1-4步无依赖，可并行推进。从哪步开始？"

👤: 开始第1步

🤖: [更新第1步 status: in_progress]
    "第1步已标记为进行中。目标：梳理当前系统架构……"
```

**Session 3 — 几天后续接，多步就绪待审阅**

```
👤: /project

🤖: [扫描 → 1 个活跃项目]
    "加载 admin-refactor（2/6 完成）。
     
     就绪待审阅：
     ⚡ 第1步 现状分析+技术选型（产出已存在）
     ⚡ 第2步 公共组件抽离（产出已存在）
     
     进行中：
     🔵 第3步 用户管理模块迁移
     
     审阅后说「第X步完成了」即可锁定。"
```

**Session 4 — 依赖汇合点**

```
👤: /project

🤖: [扫描 → 1 个活跃项目]
    "加载 admin-refactor（4/6 完成）。
     
     已完成：✅ 第1步 ✅ 第2步 ✅ 第3步 ✅ 第4步
     已解锁：⚡ 第5步 全局状态管理迁移（依赖的第2步已完成！）
     
     第6步仍需等待第5步。
     需要加载前4步的全量产出作为上下文吗？"
```

## 适用场景

凡是需要多步骤、跨多天推进的事，都能用 `/project`：

| 场景 | 为什么适合 |
|------|-----------|
| 重构老项目（分模块渐进） | 依赖关系明确，模块可并行，天然跨天 |
| 写系列文章/教程（多篇并行） | 同一主题可并行调研，发布有先后依赖 |
| 策划+开发一个 side project | 从想法到上线，步骤拆解后跨数周执行 |
| 系统学习一个主题（研究+整理） | 多方向并行调研，最终汇合为完整产出 |
| 任何"一次会话做不完"的事 | 只要需要 2+ 次会话，就值得用 `/project` |

## 与其他方案对比

| 方案 | 核心机制 | 优势 | 局限 |
|------|---------|------|------|
| **/project（本方案）** | DAG 状态文件 + 依赖编排 | 并行/串行编排，多项目隔离，零依赖 | 单 Skill，功能聚焦 |
| cowork-session-skills | `/standup` + `/conclude` | 极简，两个文件 | 线性续接，无依赖概念 |
| agent-handoff-skill | 多文档 .agent-handoff/ | 跨平台，结构化 | 线性续接，手动管理 |
| CPR | /preserve /compress /resume | 压缩日志，Token 优化 | 聚焦会话日志，非项目管理 |
| handoff-mcp | MCP Server + 25+ 工具 | 任务树 + 里程碑 | 需安装 MCP Server |

**`/project` 的差异点：不做会话日志记录，做项目 DAG 编排。** 配合任何会话日志 Skill 使用更佳。

## 常见问题

### 和 CLAUDE.md 的 auto-memory 有什么区别？

CLAUDE.md 的 auto-memory 记录的是"事实"（用了什么库、决定了什么架构）。`/project` 管理的是"进度"（步骤 A 完成、步骤 B 等待步骤 A、产出文件在哪个路径）。

两者互补：CLAUDE.md 存知识，`/project` 存进程。

### 我能同时推进多个项目吗？

可以。每个项目一个 YAML 文件，完全隔离。`/project` 在有多个时会让你选。

### 怎么把项目归档？

说"归档 <项目名>"。状态文件会移到 `.claude/projects/archive/`，不删除，可随时恢复。

### 产出的文件太多，上下文会爆吗？

不会。`/project` 只加载 L1（context，~200字）+ L2（每步一行摘要）。产出文件全文只在你说"看看 XX"时才加载。

### 我用过其他方案，能迁移吗？

可以。在 `.claude/projects/` 下手动创建状态文件，填入已有步骤即可。格式见 `templates/project-state.yaml` 和 `references/schema.md`。

```

├── <项目名>.yaml          ← 状态文件（DAG + 进度 + 会话记录）

```
