# /project — 跨会话项目管理 Skill

[![License: MIT](https://img.shields.io/badge/License-MIT-blue.svg)](LICENSE)
[![Claude Code](https://img.shields.io/badge/Claude%20Code-Skill-orange)](https://claude.ai/code)

**一个 Skill，一条命令，让 Claude Code 拥有跨会话推进复杂项目的能力。**

---

## 为什么需要这个？

Claude Code 没有跨会话记忆。每次新会话，Claude 从零开始——不知道你之前做了什么、哪些步骤已完成、卡在哪里、产出过什么结论。

现有社区方案大多解决"线性续接"问题（上次做到哪了，下次继续）。但实际项目往往是**多步骤、有依赖、可并行的 DAG 结构**——你需要在不同会话完成不同步骤，最终汇合。

`/project` 解决的就是这个：**不是记笔记，是编排项目。**

---

## 快速开始

### 安装

```bash
mkdir -p ~/.claude/skills/project
cp SKILL.md ~/.claude/skills/project/
```

或使用初始化脚本：

```bash
bash scripts/init.sh
```

### 第一次使用

```
你: "我想做一个博客系统，前后端分离，支持 Markdown，部署到 Vercel。"

Claude: [讨论...分析...拆解为 4 步，识别依赖关系]

你: "可以，确认这个计划。"

Claude: "计划已明确。输入 /project 保存为跨会话项目。"

你: /project myblog

Claude: [计划已保存]
       "下次在任何会话输入 /project 即可继续。
        当前 4 步，预计 2-3 次会话。从第 1 步开始？"

你: 开始吧

--- [下次会话] ---

你: /project

Claude: [读状态文件 → 加载上下文 → 汇报进度]
       "myblog · 进行中。第 1 步已完成。
        第 2 步已解锁，可开始。继续？"
```

**就一条命令：`/project`。** 不记子命令，不记参数。

## 能力与边界（诚实表格）

我们对这个 Skill 的能力边界保持透明：

| ✅ 能做到 | ❌ 做不到 |
|-----------|----------|
| `/project` 唤起 → 读 YAML → 恢复项目上下文 | 在你没说 `/project` 时自动感知"该续接了" |
| 检测产出文件是否实际存在 | 记住你没写进 YAML 或产出文件的讨论细节 |
| 根据依赖关系判断哪些步骤可开始、哪些阻塞 | 自动判断产出质量是否合格 |
| 多项目并存（命名空间隔离） | 跨项目自动感知关联 |
| `/project` 加载后，后续用自然语言交互 | 在你忘记 `/project` 时强制触发 |

**核心承诺：一次 `/project`，就能在中断后的新会话里恢复项目全貌——有哪些步骤、产出什么、卡在哪、该做什么。**

## 核心卖点

### 1. DAG 编排，不只是线性笔记

```
并行组1 ─┬─ 第1步 国际化方案调研 ──────────┐
         ├─ 第2步 支付渠道整合 ──→ 第3步 ───┤
         └─ 第4步 性能基准测试 ────────────┘
                                            ↓
                                    第5步 架构汇合点
                                            ↓
                                    第6步 收尾文档
```

现有方案只记录"上次做到第3步了"。`/project` 会告诉你：哪些产出就绪待审阅、哪些因为依赖未满足而阻塞、哪些可以并行推进。

### 2. 一条命令，零子命令

不区分 `/project continue`、`/project wrap`、`/project done`。就 `/project`，后续自然语言：

| 你说（自然语言） | Skill 响应 |
|-----------------|-----------|
| `/project` | 加载上下文，汇报进度 |
| "第1步完成了" | 锁定完成状态，检查哪些步骤新解锁 |
| "今天就到这" | 更新状态，追加会话摘要 |
| "归档这个项目" | 确认后移入 archive/ |

### 3. 多项目隔离

```
.claude/projects/
  blog.yaml
  refactor.yaml
  shop-global.yaml
  archive/              ← 已完成或放弃的项目
    old-feature.yaml
```

`/project` 只有一个活跃计划时直接加载，多个时列出让你选。项目之间完全隔离，互不影响。

### 4. 上下文分级注入

全量加载产出文件会吃掉 context window。`/project` 用三级加载：

| 级别 | 加载内容 | 触发条件 |
|------|---------|---------|
| L1 | `context` 字段（200 字项目背景） | 每次 `/project` |
| L2 | 每个 done 步骤的一行摘要 | 每次 `/project` |
| L3 | 产出文件全文 | 用户说"看看 XX 产出"时 |

### 5. 零依赖

一个 `SKILL.md` 文件 + 项目里的 `.claude/projects/*.yaml`。不需要 MCP Server、Python 脚本、API key、外部服务。

## 完整演示

### 场景：电商平台多语言多币种改造（7 步，有依赖）

**Session 1 — 讨论 + 生成计划**

```
👤: "我们的电商平台要改造成多语言多币种。涉及国际化框架、
     支付渠道整合、汇率结算、性能保障...帮我梳理计划。"

🤖: [讨论目标 → 分析依赖 → 拆解步骤]

    建议 7 步：
    第1步 国际化方案调研 ─┐
    第2步 支付渠道整合 ───┤ 可并行
    第4步 性能基准测试 ───┘
    第3步 汇率+结算 ← 依赖第2步
    第5步 架构设计   ← 依赖第1-4步全部
    第6步 技术文档+迁移方案 ← 依赖第5步
    第7步 分阶段实施 ← 依赖第6步

    预计 5-7 次会话。确认后输入 /project 保存。

👤: /project shop-global

🤖: "shop-global 已保存。下次输入 /project 继续。"
```

**Session 2 — 并行工作**

```
👤: /project

🤖: [读 YAML → 加载 context]
    "shop-global · 电商多语言改造
     
     进度：
     ⬜ 第1步 国际化方案调研
     ⬜ 第2步 支付渠道整合
     ⬜ 第4步 性能基准测试
     🔴 第3步（等第2步）
     🔴 第5步（等1+2+3+4）
     
     第1、2、4步可并行推进。从哪步开始？"

👤: 第1步

🤖: [开始国际化方案调研...产出调研报告]
    "第1步完成，产出：docs/internationalization-survey.md
     审阅后说「第1步完成了」即可锁定。"
```

**Session 3 — 续接 + 锁定**

```
👤: /project

🤖: [读 YAML → 检测产出文件]
    "shop-global
     
     就绪待审阅：
     ⚡ 第1步 国际化方案调研（产出已存在）
     ⚡ 第4步 性能基准测试（产出已存在）
     
     进行中：
     🔵 第2步 支付渠道整合
     
     审阅后说完成即可推进。"
```

**Session 4 — 汇合点**

```
👤: /project

🤖: [读 YAML → 检查依赖]
    "shop-global
     
     已完成：✅ 第1步 ✅ 第2步 ✅ 第3步 ✅ 第4步
     已解锁：⚡ 第5步 架构设计（全部依赖满足！）
     
     需要加载前4步的全量产出作为上下文吗？"
```

## 与其他方案对比

| 方案 | 核心机制 | 优势 | 局限 |
|------|---------|------|------|
| **/project（本方案）** | DAG 状态文件 + 依赖编排 | 并行/串行编排，多项目隔离 | 单 Skill，功能聚焦 |
| cowork-session-skills | `/standup` + `/conclude` | 极简，两个文件 | 线性续接，无依赖概念 |
| agent-handoff-skill | 多文档 .agent-handoff/ | 跨平台，结构化 | 线性续接，手动管理 |
| CPR | /preserve /compress /resume | 压缩日志，Token 优化 | 聚焦会话日志，非项目管理 |
| handoff-mcp | MCP Server + 25+ 工具 | 任务树 + 里程碑 | 需安装 MCP Server |

**`/project` 的差异点：不做会话日志记录，做项目 DAG 编排。** 配合任何会话日志 Skill 使用更佳。

## 常见问题

### 和 CLAUDE.md 的 auto-memory 有什么区别？

CLAUDE.md 的 auto-memory 记录的是"事实"（用了什么库、决定了什么架构）。`/project` 管理的是"进度"（步骤 A 完成、步骤 B 等待步骤 A、产出文件在哪个路径）。

两者互补：CLAUDE.md 存知识，`/project` 存进程。

### 我能同时推进多个项目吗？

可以。每个项目一个 YAML 文件，完全隔离。`/project` 在有多个时会让你选。

### 怎么把项目归档？

说"归档 <项目名>"。状态文件会移到 `.claude/projects/archive/`，不删除，可随时恢复。

### 产出的文件太多，上下文会爆吗？

不会。`/project` 只加载 L1（context，~200字）+ L2（每步一行摘要）。产出文件全文只在你说"看看 XX"时才加载。

### 我用过其他方案，能迁移吗？

可以。在 `.claude/projects/` 下手动创建状态文件，填入已有步骤即可。格式见 `templates/project-state.yaml` 和 `references/schema.md`。

## 文件结构

```
~/.claude/skills/project/
├── SKILL.md              ← 核心 Skill 指令
└── (本仓库)

项目中的文件：
.claude/projects/
├── <项目名>.yaml          ← 状态文件（DAG + 进度 + 会话记录）
└── archive/               ← 归档项目
```

## 适用场景

| 场景 | 举例 |
|------|------|
| 多步骤开发项目 | 重构系统 → 5 步，3 步可并行 |
| 产品规划 + 实施 | 先讨论需求，分步落地 |
| 多阶段迁移 | 数据库迁移分 N 批执行 |
| 研究 + 写作 | 多主题并行研究，最后汇合 |
| 任何"一次会话做不完"的事 | 所有需要 2+ 次会话的项目 |
